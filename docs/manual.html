Qua Manual

Manuel J. Simoni

Qua is a new dialect of Lisp, designed to be exceptionally simple and allow tiny implementations, yet offer the powerful metaprogramming facilities, control flow abstractions, and general no-nonsense approach that Lisp programmers have come to expect. Qua is based on Kernel, Common Lisp, and Scheme. From Kernel it takes its central computing workhorses, lexically-scoped fexprs and first-class environments. The surface syntax and core language look and feel a lot like Common Lisp, from which Qua inherits many operators. The interface for control flow manipulation, delimited continuations, is the result of a long line of research pioneered in Scheme.

Qua is still unfinished. Especially the areas of strings, numbers, objects, classes, generic functions, methods, sequences, streams, and the JS interface are still in flux and/or unspecified. Nevertheless, it is already a practical language and the functionality documented in this manual is not expected to change in major ways.

Syntax

Apart from some extra syntax for JS-related functionality, Qua syntax mostly follows common Lisp syntactic conventions and shouldn't be too surprising.

String Syntax (Syntax)

Syntax:

"characters" \" \t \r \n \\

Description:

Syntax for strings follows [JavaScript Object Notation] but only supports a subset of it at the moment.

Examples:

"foo \" bar" => "foo \" bar"

Rationale:

Given the close integration between Qua and JavaScript it seems to be a good idea to follow JavaScript's string syntax, and not more traditional Lisp syntax, so that programmers don't need to keep two syntaxes in their heads.

Number Syntax (Syntax)

Syntax:

[+|-]digits[.digits]

Description:

Syntax for numbers follows [JavaScript Object Notation] but only supports a subset of it at the moment.

Examples:

-12.34 => -12.34

Rationale:

See rationale for String Syntax.

Constant Syntax (Syntax)

Syntax:

#constant

Description:

Built-in constants, such as #T or #VOID get a special syntax distinct from symbols.

Examples:

#t => #t
#void => #void

Rationale:

Avoiding to pollute the Variable Namespace with identifiers for constants, which should be short, seems to be a good idea.

Variable Symbol Syntax (Syntax)

Syntax:

symbol-name

Description:

Variable symbols allow many characters except space (exact list to be determined) and should allow all alphanumeric Unicode characters at some point, but are currently restricted to ASCII.

Examples:

foo-bar*

Rationale:

This is just the usual Lisp symbol syntax.

Function Symbol Syntax (Syntax)

Syntax:

#'function-name

Description:

Function symbols follow Variable Symbol Syntax as to the content of the symbol name.

Examples:

#'+
#'quuxify

;; Unlike Common Lisp, Qua universally allows function symbols
;; as definiends and parameters:
(def #'my-function (lambda () 1))
(my-function) => 1

(defun compose (#'f #'g)
  (lambda (x) (f (g x))))

Rationale:

While Qua handles namespaces differently than existing Lisps, it makes sense to keep the familiar syntax for referring to functions, and more generally, operators.

Keyword Symbol Syntax (Syntax)

Syntax:

:keyword-name

Description:

Keyword symbols follow Variable Symbol Syntax as to the content of the symbol name.

Examples:

:my-keyword => :my-keyword

Rationale:

This is just the usual Lisp keyword syntax.

List Syntax (Syntax)

Syntax:

( element* )

Description:

The usual syntax for lists.

Examples:

(a big (nested (list)))

() === #nil

Rationale:

Proven syntax since 1958.

Dotted List Syntax (Syntax)

Syntax:

( element+ . element )

Description:

The usual syntax for specifying the last element of a List explicitly.

Examples:

(a dotted . list)

(1 2) === (1 . (2)) === (1 . (2 . #nil))

Rationale:

Proven syntax since 1958.

Quotation Syntax (Syntax)

Syntax:

'form

Description:

The usual syntax for preventing evaluation of a Form, syntactic sugar for QUOTE.

Examples:

'foo => foo
'#'foo => #'foo
'12 => 12
''foo => 'foo
'(+ 3 4) => (+ 3 4)

Rationale:

Proven syntax since 1958.

Dynamic Variable Naming Convention (Naming Convention)

Syntax:

*dynamic-name*

Description:

Names of dynamic variables are wrapped in ``*'' by convention.

Examples:

*standard-output*

Rationale:

Using special prefixes and suffixes prevents having to use prefixes like ``current-'' for dynamic variable names to distinguish them from local variables.

Constant Variable Naming Convention (Naming Convention)

Syntax:

+constant-name+

Description:

Names of constant variables are wrapped in ``+'' by convention. This should be used for variables that are never reassigned, and that are bound to objects whose contents are immutable.

Examples:

(defconstant +my-constant+ 23)

Rationale:

Common Lisp convention.

Global Variable Naming Convention (Naming Convention)

Syntax:

-global-name-

Description:

Names of global variables are wrapped in ``-'' by convention. This should be used for variables that are reassigned, like a global counter, or that are bound to objects whose contents are mutated, like a global hash table.

Examples:

(def -my-global-hash-table- (js-object))

Rationale:

It seems like a good idea to syntactically distinguish global variables, analogous to dynamic variables.

JS Global Variable Syntax (Syntax)

Syntax:

$variable

Description:

Syntax for accessing a JavaScript global variable or function. Can be used with SETF to update the variable. This is syntactic sugar for JS-GLOBAL.

Examples:

;; Access global JavaScript variables:
$window => #[js-object]

;; Call global JS functions:
($alert "Hello world!") => #void

;; Update global JavaScript variables:
(setf $SOME_JS_VARIABLE 12) => 12

Rationale:

Qua needs convenient access to JS globals, so it seems to make sense to spend a special sigil for that purpose.

JS Property Syntax (Syntax)

Syntax:

.property

Description:

Syntax for accessing a JavaScript property. Can be used with SETF to update the property. This is syntactic sugar for JS-GETTER.

Examples:

(.title $document) => "Qua Lisp Manual"

(setf (.title $document) "Qua rocks")

;; Is actually shorthand for a function, so can be mapped across lists
;; of objects, etc:
(map .x (list (js-object :x 1 :y 2) (js-object :x 3 :y 4)))
=> (1 3)

Rationale:

See rationale for JS Global Variable Syntax.

JS Method Syntax (Syntax)

Syntax:

@method

Description:

Syntax for invoking a JavaScript method. This is syntactic sugar for JS-INVOKER.

Examples:

(@toString 12) => "12"

(@log $console "This is a log message sent to the browser console")

;; Like .property syntax, this returns a function, so we can
;; do things like:
(map @toString (list 1 2 3)) => ("1" "2" "3")

Rationale:

See rationale for JS Global Variable Syntax.

Line Comment Syntax (Syntax)

Syntax:

; comment

Description:

Syntax for comments until the end of the line.

Examples:

; this is a comment

;;; this is also a comment

Rationale:

Classic Lisp.

Evaluation

Evaluation is the process of turning a Form into a Value. Evaluation happens either implicitly, e.g. at the REPL, or explicitly, under programmer control via EVAL.

Form (Concept)

Description:

A form is any OBJECT meant to be evaluated. It is either a Self-Evaluating Form, an Identifier Form, or a Compound Form.

Examples:

;; Self-evaluating forms:
#t => #t
12 => 12
"foo" => "foo"
:key => :key

(def x 1)
x => 1 ; Identifier form

;; Compound forms:
(+ 1 2) => 3
(if #t 1 2) => 1

Rationale:

Classic Lisp.

Self-Evaluating Form (Concept)

Description:

Objects like constants, strings, numbers, and keyword symbols are said to be self-evaluating forms, that is, they simply evaluate to themselves.

Examples:

#t => #t
12 => 12
"foo" => "foo"
:key => :key

Rationale:

Classic Lisp.

Constant (Concept)

Description:

A constant is a built-in OBJECT that is a Self-Evaluating Form. It is not possible for the Qua programmer to define new constants. Constants are distinct from constant variables, which are simply variables that are (hopefully) never modified.

Core Qua has the following constants: #NIL, #T, #F, #VOID, and #IGN. In addition, there are #NULL and #UNDEFINED from JavaScript.

Examples:

#t => #t
#void => #void

Rationale:

Classic Lisp.

Identifier Form (Concept)

Description:

A SYMBOL used as a Form is called an identifier form and evaluates to the value of the Binding identified by the symbol. An error is signaled if the symbol is unbound.

Examples:

(def x 1)
x => 1 ; Variable binding

(defun x () 1)
#'x => #[function] ; Function binding

Rationale:

Classic Lisp, with the difference that Qua treats the Variable Namespace and Function Namespace uniformly.

Compound Form (Concept)

Description:

A CONS used as a Form is called a compound form.

If the CAR of the compound form is a SYMBOL, it gets looked up in the Function Namespace.

Otherwise, the CAR gets evaluated normally.

In either case, the resulting value must be an Operator, or an error is signalled.

Finally, the CDR of the compound form is passed as Operand to the operator, and the resulting Value is returned.

Examples:

;;; Symbol as first element:

;; Even though + is a variable symbol, it gets turned into a function 
;; symbol and then looked up.
(+ 1 2) => 3

;; It is also possible, although never done in practice,
;; to specify a function symbol as first element explicitly
;; and write it like this:
(#'+ 1 2) => 3

;;; Non-symbol as first element:

;; The LAMBDA gets evaluated normally and then receives (1 2) as operands
((lambda (x y) (+ x y)) 1 2) => 3

;; Unlike Common Lisp, and like Kernel and Scheme, Qua allows any 
;; expression as the first element of a compound form, not only LAMBDA
;; expressions:
(defun my-function-returning-function ()
  (lambda (x y) (+ x y)))
((my-function-returning-function) 1 2) => 3

Rationale:

Evaluation of compound forms in Qua can be viewed as a mix of the evaluation rules of Common Lisp on the one hand, and Kernel and Scheme on the other. Like Common Lisp, Qua treats symbols used as the first element of a compound form specially, by looking them up in the Function Namespace. But like Kernel and Scheme, Qua places no restrictions on non-symbols used as the first element, leading to slightly more expressivity.

Operator (Concept)

Description:

All computation in Qua is performed by operators. All operators have in common that they receive operands and return a Value.

Operators can be classified as special operators (fexprs and macros) and functions.

Examples:

;; + evaluates to a function operator.
;; It receives the list (1 2 3 4) as operands.
;; It returns the value 10.
(+ 1 2 3 4) => 10

Rationale:

Qua operators are what Kernel calls combiners. I just couldn't get myself to use this word, so I chose operator instead, since this term is used in Common Lisp for a similar concept.

Special Operator (Concept)

Description:

A special operator is an Operator that, unlike a FUNCTION, uses special rules to determine which of its operands are evaluated.

Special operators may be fexprs or macros.

Examples:

;; IF is a special operator.  In this case,
;; the form (* 3 3) is never evaluated.
(if #t (* 2 2) (* 3 3)) => 4

Rationale:

Qua uses the term special operator to mean any operator with special evaluation rules. The reason is that users need not be aware of how a special operator is implemented, be it as a fexpr or macro, only that it has special evaluation rules.

Operand (Concept)

Description:

The operands are the data passed as input to an Operator. If the operator is a Special Operator, the operands are passed to it unevaluated. If the operator is a FUNCTION, then each operand is evaluated, and the result is called an Argument.

While the operands passed to a function must be a List, there is no such requirement for special operators: they simply receive the CDR of the Compound Form. In the most general case of the operands consisting of nested forms, we therefore speak of operand trees. An operand tree is matched against the operator's parameter tree.

Examples:

;; A fexpr that receives two unevaluated operands and
;; evaluates them explicitly:
(deffexpr a-fexpr (op1 op2) env
  (* (eval op1 env) (eval op2 env)))
(a-fexpr (+ 1 1) (+ 2 2)) => 8

;; A function that does the same thing -- its operands
;; are automatically evaluated to arguments:
(defun a-function (arg1 arg2)
  (* arg1 arg2))
(a-function (+ 1 1) (+ 2 2)) => 8

;; Unlike most Lisps, and like Kernel, Qua doesn't actually require
;; the operands of an operator to be a list.  While usually it is a
;; list, it is still worthwhile to point this out because it is a
;; fundamental feature of Kernel-like languages.  Here we pass a single
;; number as operand to a fexpr:
((vau x #ign x) . 1) => 1

;; A fexpr that destructures its operand tree (1 ((2))) using the
;; parameter tree (x ((y))):
(deffexpr weird (x ((y))) #ign
  (+ x y))
(weird 1 ((2))) => 3

Rationale:

See [The Kernel Programming Language].

Argument (Concept)

Description:

An argument is the result of evaluating an Operand.

Since arguments are simply evaluated operands, we sometimes refer to them as operands.

Examples:

;; Operands to a function are automatically evaluated to arguments.
;; Here (+ 1 1) and (+ 2 2) are the operands to the * function.  They
;; are evaluated to the arguments 2 and 4 before being passed to the
;; function.
(* (+ 1 1) (+ 2 2)) => 8

Rationale:

See [The Kernel Programming Language].

Parameter (Concept)

Description:

Parameters are the means by which operators receive and assign names to operands and arguments passed to them.

If a parameter is a SYMBOL, the operand is bound to the name of the symbol within the body of the operator.

If a parameter is a CONS, the CAR and CDR are recursively treated as parameters, giving rise to so-called parameter trees (see example below). These allow destructuring of operand trees.

If a parameter is #NIL, the operand must also be #NIL or an error is signalled.

If a parameter is #IGN, the operand is ignored (not bound to a name).

Examples:

;; A function that binds its whole argument list to a single name:
(defun return-args-list args
  args)
(return-args-list 1 2 3 4) => (1 2 3 4)
(return-args-list) => ()

;; A function that binds the first two arguments to the names A and B,
;; and the rest of the arguments to the name C:
(defun rest-args (a b . c)
  (list a b c))
(rest-args 1 2 3 4) => (1 2 (3 4))
(rest-args 1 2) => (1 2 ())

;; A function that allows no arguments (in other words, the operand
;; must be #NIL):
(defun no-args () 1)
(no-args) => 1
;; Same thing as:
(defun no-args #nil 1)

;; It is also possible to deeply destructure operands and arguments by
;; means of parameter trees.  Here, A gets bound to the first
;; argument, 1.  B gets bound to the first element of the second
;; argument (which is a list), 2.  C gets bound to the rest of the
;; second argument, (3 4).
(defun destructure (a (b . c))
  (list a b c))
(destructure 1 '(2 3 4)) => (1 2 (3 4))

;; A function that ignores everything past the first argument:
(defun ignore (a . #ign) a)
(ignore 1 2 3 4) => 1

Rationale:

See [The Kernel Programming Language].

Environment Parameter (Concept)

Description:

In addition to ordinary parameters, fexprs have a special parameter, called environment parameter, through which they receive the ENVIRONMENT in which they are called.

Examples:

;; A fexpr that returns the value that the variable X has
;; in the environment in which the fexpr is called (here the 
;; environment parameter is called ENV but it could be called
;; anything):
(deffexpr lookup-x-in-environment () env
  (eval 'x env))

(def x 1)
(lookup-x-in-environment) => 1
(let ((x 2))
  (lookup-x-in-environment)) => 2

;; The environment parameter can be ignored if unneeded:
(deffexpr other-fexpr () #ign
  12)
(other-fexpr) => 12

Rationale:

See [The Kernel Programming Language].

Value (Concept)

Description:

Value is the term for any OBJECT that is the result of the evaluation of a Form or that is the value of a Binding.

Examples:

;; The form (+ 1 1) evaluates to the value 2:
(+ 1 1) => 2

(def x 1)
x => 1 ; the variable X is bound to the value 1

Rationale:

Classic Lisp.

FEXPR (Class)

Description:

Fexprs are the most fundamental type of user-definable Operator in Qua: both functions and macros are derived from them.

A fexpr receives its operands as unevaluated forms and has to explicitly use EVAL on them if evaluation is desired.

In addition to an ordinary parameter tree, a fexpr also has an Environment Parameter through which it receives the ENVIRONMENT in which it is called.

Examples:

;; A very simple fexpr that simply returns its single operand,
;; analogous to QUOTE:
(deffexpr my-quote (operand) #ign
  operand)
(my-quote (1 2 3)) => (1 2 3)

;; A fexpr that has no ordinary parameter, but makes use of the
;; environment parameter and returns it, analogous to THE-ENVIRONMENT:
(deffexpr my-current-environment () env
  env)
(let ((x 1))
  (def current-env (my-current-environment))
  (eval 'x current-env)) => 1

;; How WHEN could be defined as a fexpr:
(deffexpr my-when (test . body) env
  (eval (list #'if test (list* #'progn body) #void) env))

(my-when (eql #t #t) 1 2 3) => 3
(my-when (eql #t #f) 1 2 3) => #void

Rationale:

See [The Kernel Programming Language].

VAU (Special Operator)

Syntax:

VAU parameter-tree environment-parameter form* => fexpr

Operands and Value:

parameter-tree---A parameter tree.

environment-parameter---An Environment Parameter.

form---A Form.

fexpr---A FEXPR.

Description:

VAU is the constructor of fexprs, analogous to how LAMBDA is the constructor of functions.

It creates a new fexpr with the given parameter tree, Environment Parameter, and body forms and returns it.

Examples:

;; Create a simple fexpr that explicitly evaluates its operands:
(def #'my-fexpr (vau (op1 op2) env 
                  (* (eval op1 env) (eval op2 env))))
(my-fexpr (+ 1 1) (+ 2 2)) => 8

Rationale:

See [The Kernel Programming Language].

DEFFEXPR (Special Operator)

Syntax:

DEFFEXPR name operand-tree environment-parameter form* => name

Operands and Value:

name---A SYMBOL.

parameter-tree---A parameter tree.

environment-parameter---An Environment Parameter.

form---A Form.

Description:

DEFFEXPR defines a named FEXPR, analogous to how DEFUN defines a named FUNCTION.

Examples:

;; Define a simple fexpr that explicitly evaluates its operands:
(deffexpr my-fexpr (op1 op2) env 
  (* (eval op1 env) (eval op2 env)))
(my-fexpr (+ 1 1) (+ 2 2)) => 8

Rationale:

While Kernel has no analogue to DEFFEXPR, Qua has it for symmetry with DEFUN.

FUNCTION (Class)

Description:

A function is an Operator that always evaluates its arguments. Inside each function is actually a FEXPR that does the computational work of the function. Argument evaluation is induced by WRAP.

Examples:

;; A simple function defined using LAMBDA:
((lambda (x y) (+ x y)) 1 2) => 3

;; LAMBDA is merely a shorthand for wrapping a FEXPR to induce
;; argument evaluation.  The above is operationally equivalent to:
((wrap (vau (x y) #ign (+ x y))) 1 2) => 1 3

Rationale:

See [The Kernel Programming Language].

WRAP (Function)

Syntax:

WRAP operator => function

Operands and Value:

operator---An Operator.

function---A FUNCTION.

Description:

WRAP constructs a FUNCTION given an underlying Operator, usually a FEXPR. All Qua functions are created by WRAP. So functions can be viewed as wrappers that induce argument evaluation for their underlying, wrapped fexpr.

Examples:

;; A simple QUOTE like-fexpr:
(def #'my-quote (vau (operand) #ign operand))
;; It returns its operands unevaluated:
(my-quote (+ 1 2)) => (+ 1 2))

;; We can WRAP the fexpr inside a function:
(def #'my-wrapped-quote (wrap #'my-quote))
;; The resulting function evaluates the operand, and passes it to MY-QUOTE:
(my-wrapped-quote (+ 1 2)) => 3

;; While usually not done, WRAP can be applied multiple times:
(def foo 12)
(my-quote 'foo) => 'foo
((wrap #'my-quote) 'foo) => foo
((wrap (wrap #'my-quote)) 'foo) => 12

Rationale:

See [The Kernel Programming Language].

UNWRAP (Function)

Syntax:

UNWRAP function => operator

Operands and Value:

function---A FUNCTION.

operator---An Operator.

Description:

UNWRAP extracts the Operator underlying a FUNCTION.

Examples:

;; Every function has an underlying operator. We can extract the
;; operator underlying the * function and call it directly:
(def #'times-operator (unwrap #'*))
(times-operator 2 4) => 8

;; Note that the following would not work, because the underlying
;; operator is a fexpr and does not evaluate its arguments (that's
;; what the wrapper does, and we have removed it):
(times-operator (+ 1 1) (+ 2 2))

Rationale:

See [The Kernel Programming Language].
