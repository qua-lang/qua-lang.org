Qua Lisp Manual

Manuel Simoni

Qua Lisp, or just Qua for short, is a new dialect of Lisp, designed to be ultra-light and allow tiny implementations, yet offer the powerful metaprogramming facilities, control flow abstractions, and general no-nonsense approach that Lisp programmers know and love. Qua is based on Kernel, Common Lisp, and Scheme. From Kernel it takes its central computing workhorses, lexically-scoped fexprs and first-class environments. The surface syntax and core language look and feel a lot like Common Lisp, from which Qua inherits many operators. The interface for control flow manipulation, delimited continuations, is the result of a long line of research pioneered in Scheme.

Qua Lisp is still unfinished. Especially the areas of strings, numbers, objects, classes, generic functions, methods, sequences, streams, and the JS interface are still in flux and/or unspecified. Nevertheless, it is already a practical language and the functionality documented in this manual is not expected to change in major ways.

Syntax

String Syntax (Syntax)

Syntax:

"characters" \" \t \r \n \\

Description:

Qua string syntax follows [JavaScript Object Notation](ref) but only supports a subset of it at the moment.

Examples:

"foo \" bar" => "foo \" bar"

Rationale:

Given the close integration between Qua and JavaScript it seems to be a good idea to follow JavaScript's string syntax, and not more traditional Lisp syntax, so that programmers don't need to keep two syntaxes in their heads.

Number Syntax (Syntax)

Syntax:

[+|-]digits[.digits]

Description:

Qua number syntax follows [JavaScript Object Notation](ref) but only supports a subset of it at the moment.

Examples:

-12.34 => -12.34

Rationale:

See rationale for String Syntax.

Constant Syntax (Syntax)

Syntax:

#constant

Description:

Built-in constants, such as #T or #VOID get a special syntax distinct from symbols.

Examples:

#t => #t
#void => #void

Rationale:

Avoiding to pollute the variable namespace with identifiers for constants, which should be short, seems to be a good idea.

Variable Symbol Syntax (Syntax)

Syntax:

symbol-name

Description:

Variable symbols allow many characters except space (exact list to be determined) and should allow all alphanumeric Unicode characters at some point, but are currently restricted to ASCII.

Examples:

foo-bar*

Rationale:

Function Symbol Syntax (Syntax)

Syntax:

#'function-name

Description:

Function symbols follow Variable Symbol Syntax as to the content of the symbol name.

Examples:

#'+
#'quuxify

Rationale:

Keyword Symbol Syntax (Syntax)

Syntax:

:keyword-name

Description:

Keyword symbols follow Variable Symbol Syntax as to the content of the symbol name.

Examples:

:my-keyword => :my-keyword

Rationale:

List Syntax (Syntax)

Syntax:

( element* )

Description:

The usual syntax for lists.

Examples:

(a big (nested (list)))

() === #nil

Rationale:

Dotted List Syntax (Syntax)

Syntax:

( element+ . element )

Description:

The usual syntax for specifying the last element of a list explicitly.

Examples:

(a dotted . list)

(1 2) === (1 . (2)) === (1 . (2 . #nil))

Rationale:

Quotation Syntax (Syntax)

Syntax:

'form

Description:

The usual syntax for preventing evaluation of a form, syntactic sugar for QUOTE.

Examples:

'foo => foo
'#'foo => #'foo
'12 => 12
''foo => 'foo
'(+ 3 4) => (+ 3 4)

Rationale:

Dynamic Variable Syntax (Syntax)

Syntax:

*dynamic-name*

Description:

Dynamic variable names are wrapped in ``*'' by convention.

Examples:

*standard-output*

Rationale:

Using special prefixes and suffixes prevents having to use prefixes like ``current-'' for dynamic variable names to distinguish them from local variables.

Constant Variable Syntax (Syntax)

Syntax:

+constant-name+

Description:

Constant variable names are wrapped in ``+'' by convention. This should be used for global variables that are never reassigned, and that are bound to objects whose contents are immutable.

Examples:

(defconstant +my-constant+ 23)

Rationale:

Global Variable Syntax (Syntax)

Syntax:

-global-name-

Description:

Global variable names are wrapped in ``-'' by convention. This should be used for global variables that are reassigned, like a global counter, or that are bound to objects whose contents are mutated, like a global hash table.

Examples:

(def -my-global-hash-table- (js-object))

Rationale:

It seems like a good idea to syntactically distinguish global variables, analogous to dynamic variables.

JS Global Variable Syntax (Syntax)

Syntax:

$variable

Description:

Syntax for accessing a JavaScript global variable or function. Can be used with SETF to update the variable. This is syntactic sugar for JS-GLOBAL.

Examples:

$window => #[js-object]

($alert "Hello world!") => #void

(setf $SOME_JS_VARIABLE 12) => 12

Rationale:

Qua needs convenient access to JS globals, so it seems to make sense to spend a special sigil for that purpose.

JS Property Syntax (Syntax)

Syntax:

.property

Description:

Syntax for accessing a JavaScript property. Can be used with SETF to update the property. This is syntactic sugar for JS-GETTER.

Examples:

(.title $document) => "Qua Lisp Manual"

(setf (.title $document) "Qua rocks")



(map .x (list (js-object :x 1 :y 2) (js-object :x 3 :y 4)))
=> (1 3)

Rationale:

JS properties are accessed frequently so it makes sense to spend a sigil for that purpose.

JS Method Syntax (Syntax)

Syntax:

@method

Description:

Syntax for invoking a JavaScript method. This is syntactic sugar for JS-INVOKER.

Examples:

(@toString 12) => "12"

(@log $console "This is a log message sent to the browser console")

Rationale:
